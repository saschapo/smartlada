## 0. Контекст проекта (кратко)

Умный светильник из заднего фонаря ВАЗ-2106. 4 автомобильные лампы накаливания 12 В
управляются через 4 модуля **D4184** (N-канальный MOSFET, low-side PWM, logic-level,
открывается от 3.3 В). Финальная прошивка будет на **ESP32-C6** (Zigbee, привязка к
Яндекс Станции Миди, 4 dimmable-light эндпоинта). **Этот документ — НЕ финальная прошивка,
а калибровочный стенд на ESP8266**, пока C6 в доставке.

## 1. Назначение и границы

**[Цель]** Проверить «железную» часть управления 4 лампами через D4184 (PWM, low-side)
и снять числа калибровки.

**[Переносится на ESP32-C6]** слои `channels` и `config` + числа калибровки (JSON).
**[Выбрасывается на C6]** слои `web` (AP+HTTP+UI) и `platform_pwm` (ESP8266) — на C6
управление идёт через Zigbee от Алисы, PWM через LEDC.

**[ВНЕ скоупа — НЕ добавлять]** Wi-Fi-провижининг, MAC-в-SSID, QR, Zigbee, realtime/WebSocket,
аутентификация, OTA, внешние PWM-библиотеки, внешние CDN.

## 2. Карта каналов (подписать в коде и UI именно так)

| Idx | Секция (key) | Название | Мощность | Рассеиватель | Группа дефолтов |
|----|--------------|----------|---------|--------------|-----------------|
| CH0 | `stop` | Стоп | 21 Вт | красный | A (21 Вт) |
| CH1 | `reverse` | Задний ход | 21 Вт | прозрачный | A (21 Вт) |
| CH2 | `turn` | Поворот | 21 Вт | жёлтый | A (21 Вт) |
| CH3 | `marker` | Габарит | 5 Вт | красный | B (5 Вт) |

**[Важно]** **4 независимых канала** (массив из 4 полных наборов калибровки). Группы A/B —
ТОЛЬКО начальные дефолты, не ограничение: каждый канал редактируется отдельно. Совпадение
CH0–CH2 после калибровки — возможный результат, а не предпосылка.

Подписи в UI и JSON — по `key`/названию секции, НЕ по голым номерам (чтобы не путать при калибровке).

## 3. Архитектура (слоями, обязательна)

```
channels/      [ПЕРЕНОСИТСЯ] логика 4 каналов: gamma, min/max duty, soft-start. Чистая
               математика duty. НИ ОДНОГО #include ESP8266/Wi-Fi.
config/        [ПЕРЕНОСИТСЯ] структура настроек + (де)сериализация JSON.
platform_pwm/  [ВЫБРАСЫВАЕТСЯ] реализация PWM под ESP8266 (analogWrite*). Изолированный
               слой; на C6 заменяется на LEDC.
web/           [ВЫБРАСЫВАЕТСЯ] AP + HTTP + UI + export/import.
```

**[Критерий изоляции]** `channels` и `config` не содержат ESP8266/Wi-Fi-зависимостей
(проверяется глазами: нет `ESP8266WiFi.h`, `analogWrite` и т.п. внутри этих слоёв).

## 4. Платформа PWM

**[Решено]** Только стандартный Arduino core ESP8266: `analogWriteRange(1023)`,
`analogWriteFreq(freq)`, `analogWrite(pin, duty)`. Без внешних библиотек — цель в т.ч.
измерить поведение «голого» softPWM на 4 каналах + активной AP. Шкала duty — **10 бит (0–1023)**.

## 5. Функциональные требования

### 5.1 Сеть
- AP: SSID `SmartLada`, пароль `smartlada`, HTTP на :80.
- Без realtime: применение по изменению слайдера (HTTP-запрос на каждое изменение).
- Страница автономна (HTML/CSS/JS с устройства, без CDN).

### 5.2 Force-safe старт **[обязательно]**
1. При старте все 4 канала = **0% duty** ещё до инициализации PWM-выхода.
2. Глобальный `max_duty_cap` (дефолт **60%**) ограничивает выход ВСЕХ каналов; применяется
   поверх пер-канального `max_duty`.
3. Снятие/изменение капа — только осознанно через UI (отдельное поле + подтверждение).
4. После снятия капа переход к яркости — через soft-start ramp, без скачка в 100%.

### 5.3 Пер-канальная калибровка (4 независимых набора)
Для каждого канала: `gamma` (дефолт 2.2; 1.8–2.6), `min_duty` (0–1023), `max_duty` (0–1023),
`soft_start_ms` (дефолт 300; 0–1000).

Дефолты:
- CH0/CH1/CH2 (21 Вт): `gamma=2.2, min_duty=0, max_duty=1023, soft_start_ms=300`.
- CH3 (5 Вт): `gamma=2.2, min_duty=0, max_duty=1023, soft_start_ms=200` (меньше нить — короче инерция).

### 5.4 Глобальные параметры
- `pwm_freq_hz` — дефолт **150**, диапазон 100–1000, редактируется из UI (искать потолок softPWM).
- `max_duty_cap` — дефолт **60%**.

### 5.5 Маппинг яркости (порядок операций зафиксирован)
```
slider% (0..100)
  → gamma: g = (slider/100)^gamma
  → масштаб в [min_duty, max_duty]: duty = min_duty + g*(max_duty - min_duty)
  → кап: duty = min(duty, max_duty_cap% * 1023)
  → soft-start ramp за soft_start_ms до целевого duty
  → analogWrite(pin, duty)
```
0% на слайдере обязан давать **гарантированный физический 0** (лампа выкл), даже если `min_duty>0`.

### 5.6 Экспорт/импорт конфига
- `GET /config.json` → полный текущий конфиг: 4 канала (с key секций) + `pwm_freq_hz` + `max_duty_cap`.
  Человекочитаемый, готовый к переносу на C6.
- `POST /config` ← принимает тот же JSON, применяет.

Пример:
```json
{
  "pwm_freq_hz": 150,
  "max_duty_cap_pct": 60,
  "channels": {
    "stop":    { "gpio": 4,  "gamma": 2.2, "min_duty": 0, "max_duty": 1023, "soft_start_ms": 300 },
    "reverse": { "gpio": 5,  "gamma": 2.2, "min_duty": 0, "max_duty": 1023, "soft_start_ms": 300 },
    "turn":    { "gpio": 12, "gamma": 2.2, "min_duty": 0, "max_duty": 1023, "soft_start_ms": 300 },
    "marker":  { "gpio": 13, "gamma": 2.2, "min_duty": 0, "max_duty": 1023, "soft_start_ms": 200 }
  }
}
```

### 5.7 UI (одна страница, инлайн JS, без фреймворков)
- 4 блока, подписанных названием секции (Стоп / Задний ход / Поворот / Габарит) + idx + мощность:
  слайдер 0–100% + числовое поле; поля калибровки gamma/min_duty/max_duty/soft_start_ms.
- Глобальный блок: pwm_freq_hz, max_duty_cap (+ явная кнопка «снять кап»).
- Кнопки: «Применить», «Export JSON» (показать + выделить для копирования), «Import JSON».
- Индикация: MAC (справочно), свободный heap, текущая частота, состояние капа.

## 6. Edge-cases и предостережения (в код + README)

- **[критично] Общая земля** ESP8266 ↔ БП 12 В обязательна (раздельные источники: лампы 12 В,
  ESP8266 по USB). Без неё коммутация непредсказуема.
- **[критично] Предохранитель** slow-blow T6.3–8 А у выхода БП. Силовой стенд (68 Вт + пусковой
  бросок), не настольная игрушка.
- **Boot-safe GPIO ESP8266:** избегать GPIO0/2/15 (strapping) и GPIO16. Кандидаты GPIO4/5/12/13/14 —
  ПОДТВЕРДИТЬ по разводке HW-365A; убедиться, что при boot каналы D4184 не открываются (pulldown
  затвора на плате; если его нет — добавить внешний).
- **Полярность D4184:** на первом включении подтвердить, что duty 0 = лампа выключена (не инверсия).
- **softPWM + AP:** при активных HTTP-запросах возможен джиттер яркости — это **ожидаемый объект
  проверки, не баг**. Зафиксировать: мерцало/нет и при какой частоте.
- **Пусковой ток накаливания:** soft-start обязателен; кап снимать не скачком.

## 7. Definition of Done

1. AP `SmartLada` поднимается, страница открывается автономно (без интернета).
2. После флэша/ребута все 4 лампы выключены; `max_duty_cap=60%` активен до ручного снятия.
3. 4 слайдера независимо управляют 4 лампами; 0% = гарантированный физический выкл.
4. Пер-канальная калибровка (gamma/min/max/soft-start) применяется и визуально влияет; каналы независимы.
5. `pwm_freq_hz` меняется из UI (100–1000 Гц).
6. `GET /config.json` отдаёт полный корректный конфиг с key секций; `POST /config` принимает и применяет.
7. Слои `channels`/`config` изолированы (глазами: нет ESP8266/Wi-Fi-зависимостей).
8. README: схема подключения (общая земля, предохранитель, раздельное питание), таблица
   «секция → GPIO», предупреждение о силовом стенде, замечание про джиттер softPWM.
